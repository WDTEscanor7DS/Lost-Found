================================================================================
 A. SYSTEM OVERVIEW
================================================================================

The Lost & Found Management System is a web-based application designed for
organizations (schools, offices, campuses) to manage the process of reporting,
tracking, matching, and claiming lost and found items.

MAIN PURPOSE:
  - Allow users to report lost or found items online
  - Automatically detect potential matches between lost and found items
  - Provide a secure, multi-step claim verification process
  - Enable administrators to verify, approve, and manage all item records
  - Generate official PDF claim receipt documents
  - Send email notifications at every stage of the workflow

The system operates on a two-role model:
  - USERS: Report items, browse found items, claim ownership, track status
  - ADMINS: Verify submissions, manage matches, approve claims, generate reports

TARGET ENVIRONMENT:
  - Local development server (XAMPP) or any PHP-capable web server
  - MySQL database backend
  - Gmail SMTP for email notifications

     - Context-aware description hints based on selected category
     - Conditional room number input for room-based locations

     - Submit a found item report with the same fields as lost items
     - Items enter a verification queue before becoming active

  4. TRACK STATUS
     - View all personally submitted items (lost and found)
     - Filter by category, status, verification status, and sort order
     - See real-time status: open, matched, or claimed

  5. BROWSE FOUND ITEMS
     - View a card-grid catalog of all available (open + approved) found items
     - Search by keyword and filter by category
     - Click "Claim This Item" to start the ownership claim process

  6. CLAIM AN ITEM
     - Fill in a detailed claim form proving ownership
     - Provide: description of the item, unique identifiers (serial numbers,
       marks, stickers), proof photo, valid ID, proof of ownership document
     - System calculates a confidence score (0-100%) based on provided details
     - Duplicate claim prevention (one active claim per user per item)

  7. MY CLAIMS
     - View all submitted claims with visual status tracker
     - Step-by-step progress: Submitted -> Under Review -> Approved
     - Download PDF claim receipt for approved claims
     - View rejection reasons if a claim is denied

  8. NOTIFICATIONS
     - System notification center with read/unread indicators
     - Match activity feed showing matches involving your items
     - Mark all notifications as read

  9. EMAIL NOTIFICATIONS
     - Receive email when a potential match is found for your lost item
     - Confirm or reject the match directly from the email
     - Receive PDF confirmation document via email after confirming
     - Receive approval/rejection emails for claims

 10. PDF DOCUMENTS
     - Download official claim confirmation PDF from the confirmation page
     - Download claim receipt PDF from My Claims page
     - PDFs include claim reference, item details, instructions, and

 B.2 ADMIN FEATURES
--------------------------------------------------------------------------------

     - Overview statistics: pending verification count, open lost items,

     - Recent items table showing the last 8 approved items

  2. ITEM VERIFICATION
     - Review newly submitted lost items (Submitted Lost Items page)
     - Review newly submitted found items (Submitted Found Items page)

  3. CATEGORY MANAGEMENT

     - Add new categories
     - Edit existing category names and status
     - Delete categories

  4. ALL ITEMS BROWSER
     - View all approved items in a searchable, filterable table
     - AJAX-powered live search with debounced input (300ms)
     - Filter by type (lost/found), category, and status
     - Edit any item's details or reset its status to open

  5. MATCHING TOOL (DETECTED MATCHES)
     - View algorithmically detected matches between lost and found items
     - See side-by-side item comparison with match score
     - Click "Item Matched" to create a match and send notification email
     - Click "Not a Match" to blacklist a pair from reappearing

     - Side-by-side comparison: claimant data vs stored item data

     - Claims Reports: All confirmed/successful matches
     - Unsolved Reports: Items with no match, sorted by age

     - Match notification emails with item images
     - Pickup instruction emails with 3-day deadline
     - Claim approval/rejection emails

 C. SYSTEM WORKFLOW (STEP-BY-STEP)
================================================================================

--------------------------------------------------------------------------------
 C.1 USER REGISTRATION & LOGIN
--------------------------------------------------------------------------------

  1. Go to the system URL (http://localhost/lost-found-main/)
  2. Click "Create an Account" to reach the registration page
  3. Enter a username, full name, and password (minimum 6 characters)
  4. The system checks for duplicate usernames and hashes the password
  5. After registration, return to the login page
  6. Enter your username and password to log in
  7. The system verifies credentials using bcrypt and redirects:
     - Admin accounts -> Admin Dashboard
     - User accounts  -> User Dashboard

  DEFAULT ACCOUNTS:
    Admin: username = "admin",  password = "admin123"
    User:  username = "User",   password = "User123"

--------------------------------------------------------------------------------
 C.2 SUBMITTING A LOST ITEM
--------------------------------------------------------------------------------

  1. Log in as a user
  2. Click "Report Lost Item" in the sidebar
  3. Fill in the form:
     - Your email address
     - Item name (e.g., "Blue Samsung Phone")
     - Category (from the dropdown: Electronics, Wallet, ID, etc.)
     - Color
     - Location where the item was lost (dropdown list)
     - Room number (if location is "Room")
     - Description (placeholder changes based on category)
     - Upload an image (optional, JPG/PNG/GIF/WEBP)
     - If category is "ID": enter ID type, number, and issuer
  4. Click Submit
  5. The item is saved with status "pending" (awaiting admin verification)
  6. You can track its status on the "Track Status" page

--------------------------------------------------------------------------------
 C.3 SUBMITTING A FOUND ITEM
--------------------------------------------------------------------------------

  1. Log in as a user
  2. Click "Report Found Item" in the sidebar
  3. Fill in the same form as lost items (with "Location Found" label)
  4. Click Submit
  5. The item enters the admin verification queue

--------------------------------------------------------------------------------
 C.4 ITEM VERIFICATION (ADMIN)
--------------------------------------------------------------------------------

  1. Admin logs in and goes to "Submitted Lost Items" or "Submitted Found Items"
  2. Reviews each pending submission
  3. Clicks "Approve" to activate the item (makes it visible and matchable)
     OR clicks "Reject" to deny the submission
  4. Approved items appear in All Items and enter the matching pool

--------------------------------------------------------------------------------
 C.5 MATCHING SYSTEM (HOW ITEMS ARE MATCHED)
--------------------------------------------------------------------------------

  The system uses a two-stage matching algorithm:

  STAGE 1 - SQL CANDIDATE GENERATION:
    - Joins all open, approved lost items with found items of the same category
    - Excludes pairs already matched or previously rejected
    - Calculates a preliminary score:
        +20 points: Same category (base)
        +10 points: Same location
        +10 points: Same color
        +15 points: Same ID type (for ID category items)
        +25 points: Same ID number (for ID category items)
    - Only pairs with preliminary score >= 15 pass to stage 2

  STAGE 2 - PHP SCORING (up to 100 points total):
    - Name similarity        (max 30 pts) - uses similar_text()
    - Description similarity  (max 15 pts) - uses similar_text()
    - Keyword overlap         (max 15 pts) - shared words after removing stop words
    - Brand detection         (max  5 pts) - checks 19 known brand names
    - Model number matching   (max 15 pts) - regex extraction and comparison
    - Color fuzzy matching    (max 10 pts) - checks 14 known colors in text
    - ID field matching       (max 20 pts) - for ID category only

    Final score is clamped to 0-100. Only matches scoring >= 60 are shown.

  ADMIN WORKFLOW:
    1. Go to "Detected Matches" page
    2. Review side-by-side item comparisons with scores
    3. Click "Item Matched" to create a match record
       -> Both items' status changes to "matched"
       -> System sends a notification email to the lost item owner
    4. Click "Not a Match" to blacklist the pair permanently

--------------------------------------------------------------------------------
 C.6 EMAIL CLAIM FLOW (FROM EMAIL TO CONFIRMATION)
--------------------------------------------------------------------------------

  After the admin creates a match:

  1. LOST ITEM OWNER receives an email titled "Your Lost Item May Have Been Found!"
     - Email shows found item details (name, category, color, location,
       description, image)
     - Contains two buttons: "YES - This is my item" and "NO - Not my item"
     - Links use a secure 64-character token (no raw IDs exposed)

  2. IF USER CLICKS "YES":
     - Redirected to the confirmation page (user_confirm_match.php)
     - Token is validated against the database
     - Match status updates from "pending" to "user_confirmed"
     - Page displays:
         * Claimant information (name, email)
         * Found item details (name, category, color, location, description)
         * Match details (ID, score, status)
         * Next steps instructions
     - A "Download Confirmation PDF" button generates a professional PDF
     - The system automatically emails the PDF to the user as an attachment
     - The match now appears in the admin's "Claimed Items" page

  3. IF USER CLICKS "NO":
     - Redirected to the rejection page (user_reject_match.php)
     - Token is validated
     - Match record is deleted
     - Both items revert to "open" status
     - System continues searching for matches

--------------------------------------------------------------------------------
 C.7 ADMIN VERIFICATION & APPROVAL PROCESS
--------------------------------------------------------------------------------

  After a user confirms a match via email:

  1. ADMIN REVIEW (Claimed Items page - Section 1):
     - Admin sees all "user_confirmed" matches
     - Reviews side-by-side item comparison
     - Options:
       a) "CONFIRM MATCH" -> Sets pickup_deadline (+3 days), sends pickup
          instruction email to user with deadline, location, ID requirements
       b) "REJECT MATCH" -> Reopens both items, sends rejection email

  2. PICKUP PERIOD (Claimed Items page - Section 2):
     - Shows all "confirmed" matches awaiting physical pickup
     - Displays countdown timer until pickup deadline
     - Urgent styling when less than 1 day remaining
     - Options:
       a) "CONFIRM CLAIMED" -> Items marked as "claimed" (process complete)
       b) "ITEM NOT CLAIMED" -> Reopens both items (user did not pick up)
     - Auto-expiry: overdue pickups are automatically rejected on page load

  3. CLAIM COMPLETION:
     - Successfully claimed items appear in the Claims Reports page
     - The full process is complete

--------------------------------------------------------------------------------
 C.8 ALTERNATIVE CLAIM FLOW (DIRECT CLAIM VIA BROWSE)
--------------------------------------------------------------------------------

  Users can also claim items without waiting for the matching algorithm:

  1. User goes to "Browse Found Items"
  2. Browses the card catalog or searches for a specific item
  3. Clicks "Claim This Item" on a found item
  4. Fills the claim verification form:
     - Name, email, phone number
     - Detailed item description
     - Unique identifiers (serial numbers, markings, etc.)
     - Proof photo, valid ID, and proof of ownership (optional uploads)
  5. System calculates a confidence score (0-100%) comparing the claim
     against stored item data:
       - Name similarity:   max 20 points
       - Color match:       max 15 points
       - Description match:  max 25 points
       - Location match:    max 10 points
       - Unique identifiers: max 15 points
       - Proof documents:   max 15 points
  6. Claim is saved with an ID like "CLM-20260320-0001"
  7. All admins and the claimant receive in-app notifications

  ADMIN PROCESSES THE CLAIM (Claim Requests page):
  8. Admin reviews the claim with a side-by-side view
  9. Views the confidence score breakdown with progress bars
  10. Views uploaded proof documents
  11. Decides:
      a) "Mark Under Review" -> Needs more investigation
      b) "Approve" (only if score >= 40%) -> Item marked "claimed",
         other pending claims auto-rejected, approval email sent
      c) "Reject" -> Rejection email sent with admin notes
  12. Approved claims can generate a PDF receipt from the admin panel

================================================================================
 E. DATABASE STRUCTURE
================================================================================

--------------------------------------------------------------------------------
 E.1 TABLE: users
--------------------------------------------------------------------------------
  PURPOSE: Stores user accounts for authentication

  COLUMNS:
    id          INT(11)         Primary Key, Auto Increment
    username    VARCHAR(100)    Unique, Not Null
    fullname    VARCHAR(200)    Nullable
    password    VARCHAR(255)    Not Null (bcrypt hash)
    role        ENUM            'admin' or 'user' (default: 'user')

  NOTE: No email column. User emails are stored per-item in the items table.

--------------------------------------------------------------------------------
 E.2 TABLE: categories
--------------------------------------------------------------------------------
  PURPOSE: Item classification categories

  COLUMNS:
    id              INT(11)         Primary Key, Auto Increment
    category-name   VARCHAR(100)    Not Null
    status          ENUM            'Active' or 'Inactive' (default: 'Active')

  DEFAULT DATA: Electronics, Wallet, ID, Bag, Keys, Clothing, Books/Notes,
                Water Bottle, Umbrella, Others

--------------------------------------------------------------------------------
 E.3 TABLE: items
--------------------------------------------------------------------------------
  PURPOSE: Stores all reported lost and found items

  COLUMNS:
    id                  INT(11)         Primary Key, Auto Increment
    user_id             INT(11)         References users.id (nullable)
    type                ENUM            'lost' or 'found'
    name                VARCHAR(100)    Item name
    email               VARCHAR(100)    Reporter's email address
    category            VARCHAR(50)     Item category
    color               VARCHAR(50)     Item color
    location            VARCHAR(100)    Where item was lost/found
    description         TEXT            Detailed description
    image               VARCHAR(255)    Uploaded image filename
    status              ENUM            'open', 'matched', or 'claimed'
    verification_status ENUM            'pending', 'approved', or 'rejected'
    id_type             VARCHAR(50)     For ID cards: type of ID
    id_number           VARCHAR(50)     For ID cards: ID number
    id_issuer           VARCHAR(100)    For ID cards: issuing authority
    date_created        TIMESTAMP       Auto-set on creation

  INDEXES:
    idx_user_id           On user_id
    idx_match_candidates  On (type, status, verification_status, category)
    ft_name_desc          FULLTEXT on (name, description)

  STATUS FLOW:  open -> matched -> claimed

--------------------------------------------------------------------------------
 E.4 TABLE: matches
--------------------------------------------------------------------------------
  PURPOSE: Records detected matches between lost and found items

  COLUMNS:
    id              INT(11)         Primary Key, Auto Increment
    lost_item_id    INT(11)         References items.id (type='lost')
    found_item_id   INT(11)         References items.id (type='found')
    match_score     INT(11)         Algorithm confidence (0-100)
    status          ENUM            'pending', 'user_confirmed', 'confirmed',
                                    or 'rejected'
    pickup_deadline DATETIME        Deadline for physical pickup (3 days)
    claim_token     VARCHAR(64)     Unique secure token for email links
    date_matched    TIMESTAMP       Auto-set on creation

  UNIQUE KEYS:
    idx_match_pair    On (lost_item_id, found_item_id)
    idx_claim_token   On claim_token

  STATUS FLOW:  pending -> user_confirmed -> confirmed
                            (or rejected at any stage)

--------------------------------------------------------------------------------
 E.5 TABLE: rejected_matches
--------------------------------------------------------------------------------
  PURPOSE: Blacklist of item pairs dismissed by admin ("Not a Match")

  COLUMNS:
    id              INT(11)         Primary Key, Auto Increment
    lost_item_id    INT(11)         References items.id
    found_item_id   INT(11)         References items.id
    rejected_date   TIMESTAMP       Auto-set on creation

  UNIQUE KEY: pair_unique On (lost_item_id, found_item_id)

  NOTE: Prevents the same pair from reappearing in the matching algorithm.

--------------------------------------------------------------------------------
 E.6 TABLE: claims
--------------------------------------------------------------------------------
  PURPOSE: Stores user ownership claims submitted via the claim form

  COLUMNS:
    id                  INT(11)         Primary Key, Auto Increment
    claim_id            VARCHAR(20)     Unique claim reference (CLM-YYYYMMDD-NNNN)
    item_id             INT(11)         References items.id
    user_id             INT(11)         References users.id
    claimant_name       VARCHAR(200)    Full name of claimant
    claimant_email      VARCHAR(100)    Claimant email address
    claimant_phone      VARCHAR(20)     Phone number (optional)
    item_description    TEXT            Claimant's description of the item
    unique_identifiers  TEXT            Serial numbers, markings, etc.
    proof_image         VARCHAR(255)    Uploaded proof photo filename
    id_document         VARCHAR(255)    Uploaded ID document filename
    proof_document      VARCHAR(255)    Uploaded proof of ownership filename
    confidence_score    INT(11)         System-calculated score (0-100)
    status              ENUM            'pending', 'under_review', 'approved',
                                        or 'rejected'
    admin_notes         TEXT            Admin's decision notes
    date_claimed        TIMESTAMP       Auto-set on creation

  INDEXES:
    claim_id (UNIQUE), idx_claim_item (item_id), idx_claim_user (user_id)

  STATUS FLOW:  pending -> under_review -> approved
                             (or rejected at any stage)

--------------------------------------------------------------------------------
 E.7 TABLE: notifications
--------------------------------------------------------------------------------
  PURPOSE: In-app notification system for users

  COLUMNS:
    id              INT(11)         Primary Key, Auto Increment
    user_id         INT(11)         References users.id
    title           VARCHAR(200)    Notification title
    message         TEXT            Full message content
    type            ENUM            'info', 'success', 'warning', or 'danger'
    is_read         TINYINT(1)      0 = unread, 1 = read
    link            VARCHAR(255)    Optional link to related page
    date_created    TIMESTAMP       Auto-set on creation

  INDEX: idx_notif_user On (user_id)

--------------------------------------------------------------------------------
 E.8 TABLE RELATIONSHIPS DIAGRAM
--------------------------------------------------------------------------------

  users (1) -----> (many) items          via items.user_id
  users (1) -----> (many) claims         via claims.user_id
  users (1) -----> (many) notifications  via notifications.user_id
  items (1) -----> (many) claims         via claims.item_id
  items (1:lost)  -----> (many) matches  via matches.lost_item_id
  items (1:found) -----> (many) matches  via matches.found_item_id

  NOTE: Foreign key constraints are NOT formally declared in the schema.
        Referential integrity is enforced at the application level in PHP.

 F. HOW TO RUN THE SYSTEM
================================================================================

--------------------------------------------------------------------------------
 F.1 PREREQUISITES
--------------------------------------------------------------------------------

  - XAMPP installed (version 8.0 or higher recommended)
    Download: https://www.apachefriends.org/
  - Web browser (Chrome, Firefox, Edge, etc.)
  - Internet connection (for Gmail SMTP email features)

--------------------------------------------------------------------------------
 F.2 STEP-BY-STEP SETUP
--------------------------------------------------------------------------------

  STEP 1: INSTALL XAMPP
    - Download and install XAMPP on your computer
    - Default installation path: C:\xampp (Windows)

  STEP 2: START SERVICES
    - Open the XAMPP Control Panel
    - Click "Start" next to Apache
    - Click "Start" next to MySQL
    - Both should show green "Running" status

  STEP 3: PLACE PROJECT FILES
    - Copy the entire "lost-found-main" folder into:
        C:\xampp\htdocs\
    - Final path should be: C:\xampp\htdocs\lost-found-main\

  STEP 4: CREATE THE DATABASE
    - Open your browser and go to: http://localhost/phpmyadmin/
    - Click "New" in the left sidebar
    - Enter database name: lost_found
    - Click "Create"
    - Click the "Import" tab at the top
    - Click "Choose File" and select:
        C:\xampp\htdocs\lost-found-main\backend\database.sql
    - Click "Go" to import (creates all tables with default data)

  STEP 5: INSTALL PHP DEPENDENCIES
    - Open a terminal/command prompt in the project folder
    - Run: php composer.phar install
    - This installs DomPDF (required for PDF generation)
    - If Composer is installed globally, you can also use: composer install

  STEP 6: VERIFY AUTO-MIGRATIONS
    - The system auto-creates additional tables (claims, notifications)
      and columns (user_id, pickup_deadline, claim_token) on first access
    - No manual action needed; just visit the site

  STEP 7: ACCESS THE SYSTEM
    - Open your browser
    - Go to: http://localhost/lost-found-main/

  STEP 8: LOG IN
    - Use the default accounts:

        ADMIN:    Username: admin    Password: admin123
        USER:     Username: User     Password: User123

    - Or click "Create an Account" to register a new user

  Registration:         http://localhost/lost-found-main/register.php
  Admin Dashboard:      http://localhost/lost-found-main/admin/

 F.4 EMAIL CONFIGURATION

================================================================================
 G. SECURITY FEATURES
================================================================================

--------------------------------------------------------------------------------
 G.1 AUTHENTICATION & SESSION MANAGEMENT
--------------------------------------------------------------------------------

  - Passwords hashed with bcrypt via password_hash(PASSWORD_DEFAULT)
  - Password verification via password_verify() (timing-safe comparison)
  - Session regeneration on login (session_regenerate_id(true)) to prevent
    session fixation attacks
  - Three authentication guard files enforce access control:
      guard_admin.php     - Admin pages only
      guard_admin_api.php - Admin API endpoints (JSON 403 for AJAX)
      guard_user.php      - Authenticated user pages
  - Guards check both $_SESSION['id'] and $_SESSION['role']
  - Session destruction on logout (session_unset + session_destroy)

--------------------------------------------------------------------------------
 G.2 SQL INJECTION PREVENTION
--------------------------------------------------------------------------------

  - ALL user-input queries use MySQLi prepared statements with bind_param()
  - Parameter types explicitly declared (s = string, i = integer)
  - No raw user input is ever concatenated into SQL queries
  - Input values are cast to appropriate types (int cast for IDs)

--------------------------------------------------------------------------------
 G.3 CROSS-SITE SCRIPTING (XSS) PREVENTION
--------------------------------------------------------------------------------

  - All output uses htmlspecialchars() before rendering in HTML
  - Database values are escaped on display, not on input
  - Email content is HTML-escaped with htmlspecialchars()

--------------------------------------------------------------------------------
 G.4 FILE UPLOAD SECURITY
--------------------------------------------------------------------------------

  Item image uploads (save.php):
    - File extension whitelist: jpg, jpeg, png, gif, webp
    - Files renamed with timestamp + random_int() (no user-controlled names)
    - Stored outside the web application's PHP execution path

  Claim document uploads (submit_claim.php):
    - File extension whitelist: jpg, jpeg, png, pdf
    - Maximum file size: 5MB
    - MIME type verification using finfo (not just extension)
    - Files renamed with random identifiers
    - Stored in uploads/claims/ directory

--------------------------------------------------------------------------------
 G.5 CLAIM VERIFICATION SECURITY
--------------------------------------------------------------------------------

  - Email claim links use 64-character cryptographic tokens generated with
    bin2hex(random_bytes(32)) - no raw database IDs exposed in URLs
  - Tokens are unique per match (UNIQUE constraint in database)
  - Token validation in both confirm and reject pages
  - PDF generation requires valid token (no session needed for email access)
  - PDF DomPDF options: isPhpEnabled=false, isRemoteEnabled=false
    (prevents code execution and remote file inclusion in PDFs)
  - Duplicate claim prevention: one active claim per user per item
  - Minimum confidence score (40%) required for admin approval

--------------------------------------------------------------------------------
 G.6 ACCESS CONTROL
--------------------------------------------------------------------------------

  - Role-based access: admin pages are inaccessible to regular users
  - Backend API endpoints are guarded against unauthorized access
  - PDF downloads require ownership verification (claim owner or admin)
  - Referer validation on item verification actions
  - Status-based restrictions (e.g., cannot reject a user_confirmed match
    via the standard reject endpoint)

--------------------------------------------------------------------------------
 G.7 INPUT VALIDATION
--------------------------------------------------------------------------------

  - Email validation via filter_var(FILTER_VALIDATE_EMAIL)
  - Required field validation on all forms
  - Password minimum length (6 characters) on registration
  - Status values validated against whitelists before database updates
  - Integer casting for all ID parameters


================================================================================
 H. KNOWN ISSUES / LIMITATIONS
================================================================================

  1. FORGOT PASSWORD NOT FUNCTIONAL
     - The forgot-password.php page is a static HTML template
     - No password reset logic is implemented
     - The form action points to a non-existent file

  2. NO EMAIL COLUMN IN USERS TABLE
     - User email addresses are stored per-item (in the items table), not
       in the users table
     - This means the system relies on item records to contact users
     - If a user has not submitted any items, they have no email on file

  3. NO FOREIGN KEY CONSTRAINTS
     - Database relationships are not enforced at the SQL level
     - Referential integrity depends entirely on application logic
     - Deleting a user would leave orphaned items, claims, and notifications

  4. DUPLICATE DB CONNECTION FILES
     - misc/connect.php and backend/config.php serve the same purpose
     - Root pages (login, register) use misc/connect.php
     - All other pages use backend/config.php
     - connect.php has an unescaped error message in JS alert (minor XSS risk)

  5. AUTO-SYNC QUERIES DUPLICATED
     - admin/index.php and admin/all-items.php both run the same 3 item
       status synchronization queries on every page load
     - This creates unnecessary database overhead

  6. HARDCODED EMAIL CREDENTIALS
     - SMTP credentials are stored in plain text in config.php
     - Should be moved to environment variables or a secure config file

  7. CATEGORY CRUD USES SEPARATE DB CONNECTION
     - Category management pages (insert, edit, delete) use misc/connect.php
       instead of backend/config.php, creating inconsistency

  8. NO PAGINATION
     - Item listings, reports, and notifications have no pagination
     - Large datasets could cause slow page loads and memory issues

  9. NO CSRF PROTECTION
     - Forms do not include CSRF tokens
     - POST actions could be triggered by malicious cross-site requests

 10. USER DASHBOARD SHOWS GLOBAL COUNTS
     - User dashboard statistics show system-wide item counts, not
       the individual user's items (Track Status does show user-specific data)

 11. NO PASSWORD CHANGE FEATURE
     - Logged-in users cannot change their password
     - Only option is direct database modification

 12. IMAGE UPLOADS NOT FULLY VALIDATED
     - Item image uploads (save.php) check only file extension, not MIME type
     - Claim document uploads (submit_claim.php) do check MIME type
